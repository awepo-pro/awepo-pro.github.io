cmake

used to manage and build the c++ program. For example, there are many c++ source code, header files and third party library. CMake can manage and run the program.

download

for macbook, download cmake and make using

brew install cmake make

of course, this is c++ program, so make sure you have clang or gcc compiler!

demo1

folder structure

├── CMakeLists.txt
└── main.cpp

this is only 1 program, CMakeLists.txt is required for cmake. cmake convert the code in CMakeLists.txt into MakeFile which is for make

# CMakeLists.txt

cmake_minimum_required (VERSION 3.10.0)
project (main)

add_executable (main main.cpp)

cmake_minimum_required limit the minimum version of cmake to run this project. In this case, 3.10.0. The project name is called main, Moreover, we can specify the language and description, ie:

project (
	main
	DESCRIPTION 	"this is my first cmake project"
	LANGUAGES 		CXX
)

but those arguments are not necessary, they are just the hint for cmake to deduce the project.

add_executable (TARGET SOURCE_CODE), in this case, the target is main, although the target name (which is like a variable name) is not necessary to be the same as the project name, they are usually the same. Therefore, we can use environment variable to replace main:

add_executable (${PROJECT_NAME} main.cpp)

tips

if you are not sure the value of variable, use message() function to print them out:

message ("this is ${PROJECT_NAME}")

in this case, the message is this is main

main.cpp example

#include <iostream>

int main() {
	std::cout << "hello cmake" << endl;
}

execute cmake

the process is a bit annoying ……. be patient!

create a folder which use for store the all the output that for executing the program. Then, run the cmake to render into makefile and run makefile to generate executable file. Lastly, execute the executable file

mkdir build
cd build
cmake -S .. -B .
make
./main

cmake -S .. -B ., -S .. means the source directory is parent folder (since the all the source code, CMakeLists.txt are in parent folder). -B . specify the build directory. If we don’t provide the build directory, the current directory is default. Therefore, cmake -S .. -B . cane be simplified to cmake -S ...

For sake of simplicity, we often use configure.sh and run.sh to replace manual typing.

# configure.sh

rm -r build
mkdir build
cd build
cmake -S .. 

# run.sh

cd build
make
./main

where main is the executable file.

Therefore, whenever we have CMakeLists.txt, just need to run configure.sh and run.sh. Moreover, if there is any changes in source files, run run.sh apply the changes.

demo3

├── CMakeLists.txt
├── foo.h
├── foo.cpp
└── main.cpp

consider that we want to make a library with foo.h and foo.cpp and both of them are in same level with main program. then CMakeLists.txt:

cmake_minimum_required (VERSION 3.10)
project (main)

aux_source_directory (. SRCS)
add_executable (${PROJECT_NAME} ${SRCS})

we add all the source files to variable SRCS, then create executable file

demo4

├── CMakeLists.txt
├── foo
│	├── CMakeLists.txt
│	├── foo.h
│	└── foo.cpp
└── main.cpp

what if we separate them with a folder. It clearly show that directory in a static library (module). Therefore, our main target is to create a module and link it to the main.cpp

# foo/CMakeLists.txt

cmake_minimum_required (VERSION 3.10)
project (foo)

add_library (${PROJECT_NAME} foo.cpp)

CMakeLists.txt

cmake_minimum_required (VERSION 3.10)
project (main)

include_directories ("${CMAKE_CURRENT_SOURCE_DIR}/foo")
add_executable (${PROJECT_NAME} main.cpp)

add_subdirectory (foo)
target_link_library (${PROJECT_NAME} foo)

include_dreictories ask the compiler to search header file in that path, therefore, we can #include the header file directly in main.cpp

// main.cpp

#include <foo.h>

int main() {
	foo::show();
}

add_subdirectory execute the CMakeLists.txt in directory foo to make it becomes a library. Then use target_link_library to link the module foo to main

demo5

However, if we want to separete header files and source code into different level of directories, say

├── CMakeLists.txt
├── foo
│	├── include
│	│	└── foo.h
│	└── src
│	    └── foo.cpp
└── main.cpp

in other words, considering directory foo as a module (apply module concept in python), we want to import the module (foo) into main.cc. We can’t just #include the relative path of foo.h static library in main.cc since the implementation (defined in foo.cpp) of the header file is not yet found by the compiler.

we need to create a CMakeLists.txt to render the whole directory foo into static library (module).

├── CMakeLists.txt
├── foo
│	├── CMakeLists.txt
│	├── include
│	│	└── foo.h
│	└── src
│	    └── foo.cpp
└── main.cpp

# foo/CMakeLists.txt

cmake_minimum_required (VERSION 3.10)
project (foo)

include_directories (include)
aux_source_directory (src LINK_SRC)

add_library (${PROJECT_NAME} ${LINK_SRC})

in order for compiler to search header files, we need to provide the path – include_directories (include) tell the compiler to search header files in foo/include directory. aux_source_directory includes all the source file (*.cpp) to a variable LINK_SRC. Lastly, add all the source file so as to create a static library called foo.

Since we tell the path of include directory to compiler, we can use #include<foo.h> directly in foo.cpp

state the function in header file

// foo.h

namespace foo {
	void show();
};

define the function in source file

// foo.cpp

#include <foo.h>
#include <iostream>

void foo::show() {
	std::cout << "hello cmake" << '\n';
}

in top level CMakeLists.txt, we add the whole static library (module) to the source file

# CMakeLists.txt

cmake_minimum_required (VERSION 3.10.0)
project (main)

add_subdirectory (foo)
include_directories ("${CMAKE_CURRENT_SOURCE_DIR}/foo/include")

add_executable (${PROJECT_NAME} main.cpp)
target_link_libraries (${PROJECT_NAME} foo)

use the function in main program

// main.cpp

#include <foo.h>

int main() {
	foo::show();
}
